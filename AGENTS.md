# AGENTS.md

See `CLAUDE.md` for the full architecture overview and the canonical list of
commands (dev, build, test, lint, worker, D1 migrations). This file only adds
operating notes for automated agents.

## Cursor Cloud specific instructions

Standard commands live in `CLAUDE.md` / `package.json` scripts — use those.
The notes below are the non-obvious, durable gotchas for this environment.

### Node version

- The project's `engines` require Node `>= 22.22.2` (e.g. `mute-stream@4`). The
  base VM's default `node` (`/exec-daemon/node`) is older and makes
  `yarn install` fail with an "incompatible engine" error.
- nvm has `22.22.2` installed and the agent's `~/.bashrc` prepends it to `PATH`,
  so interactive shells already use the right node. The startup update script
  also forces this node version before installing. If you open a shell and
  `node -v` is < 22.22.2, run `export PATH="$HOME/.nvm/versions/node/v22.22.2/bin:$PATH"`.

### Secrets / env files (regenerated each session)

- `.env` (frontend) and `worker/.dev.vars` (worker) are git-ignored and are
  regenerated by the startup update script from injected environment-variable
  secrets. Do not commit them.
- `wrangler dev` reads worker secrets from `worker/.dev.vars` only — it does NOT
  pick up `process.env`. If you change a worker secret, rewrite
  `worker/.dev.vars` and restart `yarn worker:dev`.

### Clerk is effectively required to render the SPA

- Despite the "Clerk is optional" comment in `src/routes/__root.tsx`, the home
  layout (`src/routes/_chat.tsx`) calls `useUser()`, which throws
  "Context can only be read within the <ClerkProvider />" when
  `VITE_CLERK_PUBLISHABLE_KEY` is unset. The page then shows
  "Something went wrong".
- The update script writes a placeholder `VITE_CLERK_PUBLISHABLE_KEY` when none
  is injected; that is enough to mount `ClerkProvider` and exercise the full
  **anonymous** chat flow (chat, RAG, quota, auto-naming). Signed-in flows need
  a real publishable key matching the worker's `CLERK_SECRET_KEY`, supplied as a
  `VITE_CLERK_PUBLISHABLE_KEY` secret.

### Local D1 must be migrated before the worker is useful

- Run `yarn db:d1:migrate:local` once per fresh VM before relying on the thread
  index / per-thread ownership. It is intentionally NOT in the update script
  (migrations are kept out of startup). Local D1 state lives under
  `worker/.wrangler/state` and is ephemeral.

### Running the app

- Two servers: worker on `:8787` (`yarn worker:dev`) and the Vite SPA on `:3000`
  (`yarn dev`); or `yarn dev:all` to run both (it waits for `:8787` first).
- The browser talks directly to the worker over WebSocket, so the worker must be
  up (and `ALLOWED_ORIGINS` must include `http://localhost:3000`) before chat
  works in the UI.
